## 1. 目的と思想

### 1.1 本制度の目的

子どもに対して年俸制お小遣いを通じて**金銭管理・意思決定・運用思考**を段階的に身につけさせる。

### 1.2 ゴール

| ゴール | 説明 |
|--------|------|
| 認識転換 | お金を「もらうもの」から「管理するもの」へ |
| 思考体験 | 使う・貯める・増やす・振り返るの一連サイクル |
| ルール明文化 | 親子間の金銭ルールを明確にし、感情的な衝突を減らす |

### 1.3 本質的な位置づけ

> **本制度は投資教育・節約教育ではなく、「判断と責任の教育」である。**

---

## 2. サービス定義

### 2.1 本サービスとは

**家庭内マネールールを合意・運用・振り返りするための親子共同アプリ**

- 親子間で金銭管理のルールを設定
- 運用状況を可視化
- 定期的な振り返りを支援

### 2.2 本サービスではないもの

| 対象外 | 説明 |
|--------|------|
| 金融機関アプリ | 銀行口座や決済機能を持たない |
| 投資サービス | 実際の投資や金融商品を提供しない |
| 決済・送金サービス | 決済アプリや送金サービスではない |

### 2.3 家計簿としての役割

本サービスは親子で共有・運用する**子ども向け家計簿**としての役割を持つ。ただし、一般的な家計簿アプリとは目的と設計思想が異なる。

#### 担う役割

- 収入（年俸・家庭内案件報酬・家庭内ビジネス収益）の記録
- 支出（裁量支出）の記録
- 手元残高・親預かり残高の可視化
- 取引履歴の保存と振り返り

#### 一般的な家計簿との違い

| 観点 | 本サービス | 一般的な家計簿 |
|------|------------|----------------|
| 目的 | 使う前の判断を促す | 支出の記録 |
| 管理者 | 親子で共同 | 一方的な管理・監視 |
| 記録の扱い | 次の行動を考える材料 | 評価や叱責の材料になりがち |

#### 親子で共有する意味

| 役割 | 説明 |
|------|------|
| 親 | 管理者ではなく、ルール設計者・監査役として関与 |
| 子ども | 意思決定者として、自分の判断と結果を引き受ける |
| 共通 | 同じ情報を見ながら、金銭について対話する土台を作る |

#### 教育的な位置づけ

この家計簿的仕組みで体感的に学ぶこと：

- お金は有限であり、管理しなければなくなる
- 収支の把握が意思決定の質を左右する
- 足りなくなった場合には、回復手段を選ぶ必要がある

> **本サービスにおける家計簿は、記録のための家計簿ではなく、判断と責任を育てるための家計簿である。**

## 3. ターゲットユーザー

### 3.1 想定ユーザー像

本サービスは**親子ペア**での利用を前提とする。

#### 親（保護者）側

| ペルソナ | 特徴 |
|----------|------|
| **教育意識の高い親** | 子どもに早くからお金の判断力を身につけさせたい |
| **ルール化志向の親** | 感情的なお小遣いのやり取りを減らしたい |
| **対話重視の親** | お金について子どもと建設的に話し合いたい |

**共通の課題**
- お小遣いの渡し方に一貫性がない
- 「買って」と言われるたびに判断に迷う
- お金の話が親子間の衝突の原因になっている

#### 子ども側

| 対象年齢 | 特徴 |
|----------|------|
| 小学校低学年（6-9歳） | お金の基本概念を学び始める段階 |
| 小学校高学年（10-12歳） | 予算管理・選択の概念を理解できる段階 |
| 中学生（13-15歳） | 長期計画・運用の概念を理解できる段階 |
| 高校生（16-18歳） | 自立に向けた実践的な金銭管理が必要な段階 |

### 3.2 想定しないユーザー

| 対象外 | 理由 |
|--------|------|
| 実際の投資・資産運用をしたい人 | 本サービスは教育目的の疑似体験に限定 |
| 決済・送金機能を求める人 | 金融機能は提供しない |
| 子どもを監視・管理したい親 | 本サービスの思想と合わない |

### 3.3 利用シーン

| シーン | 内容 |
|--------|------|
| 年度初め | 年俸額・ルールを親子で合意 |
| 日常 | 支出の記録、残高確認 |
| 案件発生時 | 家庭内案件の登録・完了報告 |
| 月末・学期末 | 中間振り返り（任意） |
| 年末 | 年次レビュー・来年度計画 |

| 差別化ポイント | 内容 |

## 6. 基本構造（全世代共通）

> 本ドキュメントで記載するルールはすべて「テンプレート候補」であり、家庭ごとの状況や方針に応じて柔軟に設定・変更可能です。

### 6.1 年俸制の定義

**年俸 = 1年間で自由裁量として使える金額の上限**

| 支給方法 | 対象 |
|----------|------|
| 年1回 | 高年齢向け |
| 半年ごと | 中学生以上推奨 |
| 月割り | 低年齢向け |

### 6.2 支出カテゴリ

| カテゴリ | 負担者 | 例 |
|----------|--------|-----|
| 必須支出 | 親 | 学校指定品、制服・体操服、成長起因の買い替え |
| 裁量支出 | 子ども（年俸から） | 私服、文房具、娯楽・趣味、友人関係の支出 |

> 境界が曖昧なものは、事前に家庭ルールとして明文化する。

## 7. 年齢別ルール設計

> 以下は子どもの発達段階に基づく推奨モデルであり、厳格な規定ではありません。各家庭の事情に応じて柔軟に適用してください。

### 7.1 小学校低学年（1-3年）：体験フェーズ

| 項目 | 内容 |
|------|------|
| **目的** | お金と物の交換を理解する / 使うと減ることを体感する |
| **制度** | 月俸制（年俸の概念は導入のみ） |
| **年間目安** | 6,000〜12,000円 |
| **支出範囲** | おやつ・小物・簡単な娯楽 |

**運用ルール**
- 親が実質的な管理者
- 使用時に必ず「何に使うか」を言語化
- 貯金箱 or 疑似残高のみ

### 7.2 小学校高学年（4-6年）：管理フェーズ

| 項目 | 内容 |
|------|------|
| **目的** | 短〜中期の予算管理 / 我慢と選択の経験 |
| **制度** | 月俸制 or 半年俸制 |
| **年間目安** | 12,000〜36,000円 |
| **支出範囲** | 娯楽・私物・一部文房具 |

**運用ルール**
- 残高を可視化
- 使途を簡単に記録（理由付き）
- 年1回の振り返り必須

### 7.3 中学生：運用フェーズ

| 項目 | 内容 |
|------|------|
| **目的** | 長期視点での予算配分 / 増やすという概念の理解 |
| **制度** | 年俸制（半年支給推奨） |
| **年間目安** | 60,000〜100,000円 |
| **支出範囲** | 娯楽・私服・交際費 |

**運用ルール**
- 途中追加原則なし
- 想定外支出は救済ルール適用
- 簡易運用制度を導入

### 7.4 高校生：自立フェーズ

| 項目 | 内容 |
|------|------|
| **目的** | 生活費的思考 / 社会に出る前の最終訓練 |
| **制度** | 完全年俸制 |
| **年間目安** | 80,000〜150,000円以上（家庭方針による） |
| **支出範囲** | 娯楽・私服・昼食・一部生活費 |

**運用ルール**
- アルバイト収入との統合管理
- 年次レビュー重視
- 親は監査役に徹する

## 8. 増える仕組み（運用ルール）

### 8.1 基本原則

- 元本割れは原則なし（教育段階では）
- 増加は時間と選択の結果として発生

### 8.2 親預かり型利息（疑似預金）

親が「家庭内銀行」として機能し、子どもがお小遣いの一部または全部を自発的に親へ預けることで、疑似的な預金体験ができる。

| 特徴 | 説明 |
|------|------|
| 預入期間 | 期間中は自由に使えない（引き出しルールは家庭ごとに設定） |
| 金利 | 教育目的で設定（市場金利とは連動しない） |
| 学習目的 | 「信頼」「時間価値」「遅延報酬」を学ぶ |

### 8.3 疑似利息

- 年末残高 × 5〜15% をボーナス付与
- 利率は年齢に応じて調整
- 手元残高・親預かり残高の両方に適用可能（利率は異なる場合あり）

### 8.4 固定運用

- 一定期間触らない資金に追加ボーナス
- 例：6ヶ月固定 → +10%

### 8.5 利息設計の推奨例（年齢別）

| 対象 | 利率 | 特徴 |
|------|------|------|
| 小学校低学年 | 年利10〜20% | 短期・成功体験重視 |
| 小学校高学年 | 年利5〜10% | 期間条件付き |
| 中学生以上 | 年利3〜7% | 固定期間付き |

> 実際の銀行金利より高く設定し、「増える」体験をしやすくすることで学習効果を高める。

## 9. 資産不足・破綻時のセーフティ設計

### 9.1 想定する「資産不足」の種類

| 対象 | 状況 |
|------|------|
| 子ども側 | 年俸の使い切り、預金ロック中で流動性がない状態 |
| 親側 | 家庭内銀行として預かった金額を一時的に即時返還できない状態 |

### 9.2 子ども側が資産不足になった場合

**原則**
- 追加支給は原則なし
- 生活必須品は別枠（親負担）として確保

**回復手段（以下から選択）**
- 年末利息の放棄
- 家庭内案件による回復
- 次年度年俸からの前借り（明示的に記録）

### 9.3 親側（家庭内銀行）が不足した場合

- 親は最終的な返還責任を負うことを明文化
- 返還が遅延する場合は「返還予定日」を必ず提示
- 遅延期間中も利息計算は継続する（信頼維持のため）

### 9.4 信頼を壊さないためのUI・運用上の注意

- 不足状態は必ず可視化する
- 隠さない・誤魔化さない
- 親の裁量による帳消しは禁止

### 9.5 教育的な位置づけ

- 破綻は失敗ではなく「設計通り起きうる事象」として扱う
- 事後に必ず振り返りを行い、翌年ルールに反映する

## 10. 親ファンド・案件制度

> 「親預かり型利息（疑似預金）」は貯蓄・信託的な仕組み、「親ファンド」はリスクや成果条件を伴う投資・案件型の仕組み（貯金と投資の違いを明確に伝える）

### 10.1 親ファンド

- 子どもが親に資金を預ける
- 事前にリターン条件を明示
- 元本保証型から開始

### 10.2 家庭内案件

#### 位置づけ（お手伝い＝労働）

家庭内案件は、いわゆる「お手伝い」を労働として再定義し、**成果に対して対価が支払われる仕組み**。

| 制度 | 目的 |
|------|------|
| 年俸制 | 「管理されるお金」を学ぶ基盤 |
| 家庭内案件 | 「自分で増やすお金」を学ぶ補助制度 |

> 年俸が不足した場合の回復手段としても機能するが、年俸の代替ではない。

#### 報酬が発生する条件

以下を**すべて満たす場合**にのみ報酬対象：

- 事前に内容と成果条件が合意されている
- 実施結果が第三者（親）から見て確認可能である
- 単発または明確な完了条件を持つ

> 日常的に行う家事（片付け・洗濯・食器下げ等）は原則として報酬対象外

#### 報酬設計の考え方

- 時給制は原則採用しない
- 案件単位で報酬額を設定する
- 難易度・成果・継続性を考慮して金額を決める

> 「時間を出せばお金がもらえる」という誤解を避け、**価値と成果が対価を生む**という理解を促す。

#### 具体例

- 家の収納改善（Before / After提出）
- 家族向け調査・比較資料の作成
- マニュアル・説明書・掲示物の制作
- 一定期間継続する改善活動（条件付き）

#### 教育的な狙い

- 働くとは「頼まれたことをやる」だけではない
- 成果がなければ報酬は発生しない
- 工夫・改善・再現性が評価される

### 10.3 家庭内ビジネス（価値提供と対価の学習）

#### 位置づけ

家庭内ビジネスは「家庭内案件（労働）」とは異なり、原価・価格・需要・継続性を伴う、より発展的な学習フェーズ。

| 制度 | 学習内容 |
|------|----------|
| 年俸 | 管理と選択の訓練 |
| 家庭内案件 | 労働と成果の訓練 |
| 家庭内ビジネス | 価値提供と対価の訓練 |

> 本制度は必須ではなく、子どもの興味や発達段階に応じて任意で導入する拡張要素

#### 基本ルール

- 原則として事前申請制
  - 何を提供するか
  - 初期費用（原価）
  - 想定価格・回収方法
- 初期費用は子ども自身の年俸または案件報酬から支出
- 家族は「顧客」であり、購入・利用は自由意思
- 赤字が出た場合でも原則として補填は行わない
- 生活必需品・安全に関わるものは対象外

#### 具体例

| 例 | 内容 |
|----|------|
| 料理を振る舞う | 材料費を自己負担し、家族に任意価格で提供 |
| 家族向けレンタル | ボードゲーム・道具などを購入し、使用料を設定 |
| 制作物の提供 | 資料・イラスト・動画・説明書などを家族向けに提供 |

#### 学習できる要素

- 原価と利益の関係
- 価格設定と需要
- 顧客満足と再利用
- 失敗と改善のサイクル

#### 年齢・難易度の目安

| 対象 | レベル |
|------|--------|
| 小学校高学年 | 単発・低額・短期 |
| 中学生 | 回収計画を伴う小規模ビジネス |
| 高校生 | 継続・改善・複数回取引 |

## 16. 通知・リマインダー設計

### 16.1 通知の種類

| 種類 | タイミング | 内容 | 対象 |
|------|-----------|------|------|
| 支給通知 | 年俸支給時 | 「年俸が支給されました」 | 子 |
| 残高警告 | 残高が一定以下 | 「残高が少なくなっています」 | 子 |
| 記録促進 | 週1回（設定可） | 「今週の支出を記録しましょう」 | 子 |
| 利息付与 | 利息計算時 | 「利息が付きました」 | 子 |
| 案件完了 | 案件承認時 | 「案件が完了しました」 | 子 |
| レビュー時期 | 年末・学期末 | 「振り返りの時期です」 | 親・子 |
| 預金満期 | 固定預金満期時 | 「預金の満期です」 | 子 |

### 16.2 通知設計の原則

- **煽らない**: 不安を煽る表現は使わない
- **促す**: 行動を強制せず、きっかけを与える
- **頻度制限**: 過剰な通知は避ける（設定で調整可能）
- **親子別**: 親向け・子ども向けで文面を変える

## 18. プライバシー・セキュリティ

### 18.1 基本方針

| 原則 | 内容 |
|------|------|
| 最小限のデータ収集 | サービス運営に必要なデータのみ収集 |
| 子どものプライバシー保護 | 子どもの情報は特に慎重に扱う |
| 家庭内完結 | 他家庭との比較・共有機能は提供しない |
| 透明性 | 収集するデータと利用目的を明示 |

### 18.2 データ保護

| 項目 | 対応 |
|------|------|
| 通信暗号化 | HTTPS必須 |
| データ暗号化 | 保存データの暗号化 |
| アクセス制御 | 家族単位でのデータ分離 |
| バックアップ | 定期的なバックアップと復元機能 |
| データ削除 | アカウント削除時のデータ完全削除 |

### 18.3 子どもに関する配慮

| 項目 | 対応 |

| 広告なし | 子ども向け画面に広告を表示しない |
| 外部共有なし | 子どものデータを外部サービスと連携しない |
| 適切な表現 | 子どもの年齢に合わせたUI・メッセージ |

### 18.4 法的遵守事項（確認必要）

- 個人情報保護法
- 児童に関するプライバシー規制
- アプリストアの子ども向けアプリガイドライン

## 21. マネタイズ設計（運営が継続できる仕組み）

### 21.1 前提（売ってはいけないもの）

- 子どもの不安を煽る課金、ギャンブル的な増え方、過度な監視機能の課金化は避ける
- 家庭内の信頼を壊す動機付け（ランキングや煽り）を収益化の中心にしない

### 21.2 収益モデル候補

| モデル | 概要 | 向いている層 | 注意点 |
|--------|------|--------------|--------|
| フリーミアム＋サブスク | 基本無料、高度機能をサブスク提供 | 長期利用者 | 価格と価値のバランス |
| 買い切り | 一度の購入で家族利用可能 | ライトユーザー | 継続収益化が難しい |
| テンプレ/教材ストア | 年齢別ルール、案件集等を販売 | 教育重視層 | コンテンツの質と更新頻度 |
| ファミリープラン | 複数子ども・端末に最適化 | 多子世帯 | 価格の心理的障壁 |
| 親向けコーチング | レビューガイド、面談質問集等 | 教育熱心な親 | 監視強化にならない設計 |
| 学校/塾/自治体向け | 教育機関向けライセンス | 法人・教育機関 | サポート体制の複雑さ |
| 企業福利厚生 | 従業員家族向け提供 | 企業 | 契約・運用の複雑さ |

### 21.3 MVP段階の推奨

- **買い切り**または**低価格サブスク**を推奨
- 決済連携や金融商品は扱わない前提
- 価値は『ルール運用の継続』に置く（＝続く仕組みが価値）

| プラン | 機能 |
|--------|------|
| 無料 | 基本台帳（収入/支出/残高）＋年俸/預かり |
| Pro | テンプレ解放＋レビュー支援＋エクスポート/バックアップ |

### 21.4 価値の源泉（課金ポイント）

- テンプレの質（年齢・フェーズに応じた実践的ルール）
- 継続できるレビュー体験（振り返り支援、言語化サポート）
- 家族内合意を作る導線（対話促進ツール）
- データの可視化（レポート、履歴、エクスポート）
- 安心設計（監査ログ、復元、バックアップ）

### 21.5 NGな収益化

| NG | 理由 |
|----|------|
| 子ども向け広告・課金煽り | 信頼を壊す |
| 監視強化課金 | 親子関係を悪化させる |
| 比較機能・ランキング煽り | 家庭内の対立を誘発 |

## 22. 需要検証計画（実装前に必ず実施）

> **重要**: 需要のないサービスを作らないために、MVP開発前に以下の検証を実施する。
> スタートアップの失敗要因の42%は「市場ニーズがなかった」こと。この検証で回避する。

### 22.1 検証の目的

| 目的 | 確認すること |
|------|-------------|
| 課題の存在確認 | ターゲットユーザーが本当に「お小遣いの渡し方」に困っているか |
| 解決策への関心 | 「年俸制」「家庭内銀行」というコンセプトに魅力を感じるか |
| 支払い意思 | お金を払ってでも使いたいと思うか |
| 継続意思 | 長期的に使い続けたいと思うか |

### 22.1.1 スモークテスト（Fake Door Test）とは

リーンスタートアップの標準手法。**実際の製品を作る前に**、LPや申込フォームで需要を測定する。

| 項目 | 内容 |
|------|------|
| 目的 | 低コストで市場需要を検証し、無駄な開発を防ぐ |
| 方法 | LPに「ダウンロード/申込」ボタンを設置→クリックで「開発中、メール登録を」と誘導 |
| 測定 | 閲覧数、クリック率、登録数、価格反応 |
| 期間 | 24時間〜数週間で検証可能 |

**成功事例**

| サービス | 検証方法 | 結果 |
|----------|----------|------|
| **Dropbox** | 製品開発前にデモ動画を公開 | 登録希望者7.5万人に急増→需要確認後に開発→6億ユーザー達成 |
| **Product Hunt** | コミュニティ形成後にWeb公開 | 3万ユーザー即時獲得→急成長 |
| **Buffer** | LP + 価格ページのみ公開 | 有料プランのクリック数で需要確認→開発着手 |

### 22.2 検証フェーズ

```
Phase 1: 課題検証（1-2週間）
   ↓
Phase 2: 解決策検証（2-3週間）
   ↓
Phase 3: 支払い意思検証（2-4週間）
   ↓
判定: Go / Pivot / Stop
```

### 22.3 Phase 1: 課題検証

**目的**: ターゲットユーザーが課題を持っているか確認

#### 方法1: ユーザーインタビュー（推奨）

| 項目 | 内容 |
|------|------|
| 対象 | 小学生〜高校生の子どもを持つ親 5-10人 |
| 方法 | 30分程度の1対1インタビュー |
| 聞くこと | 現在のお小遣いの渡し方、困っていること、理想の状態 |

**インタビュー質問例**
1. お子さんへのお小遣いはどのように渡していますか？
2. お小遣いに関して困っていることはありますか？
3. お子さんとお金の話をする機会はありますか？
4. お子さんにお金について学んでほしいと思いますか？
5. 理想的なお小遣いの仕組みがあるとしたら、どんなものですか？

#### 方法2: アンケート調査

| 項目 | 内容 |
|------|------|
| 対象 | 50-100人以上 |
| 方法 | Googleフォーム等でオンライン調査 |
| 配布先 | SNS、親向けコミュニティ、知人経由 |

**判定基準（Phase 1）**
- ✅ Go: 70%以上が「お小遣いの渡し方に課題を感じている」
- ⚠️ 要検討: 50-70%が課題を感じている
- ❌ Stop: 50%未満

### 22.4 Phase 2: 解決策検証

**目的**: 本サービスのコンセプトに魅力を感じるか確認

#### 方法1: コンセプトテスト

| 項目 | 内容 |
|------|------|
| 準備物 | サービス説明資料（1-2ページ） |
| 対象 | Phase 1でインタビューした人 + 新規5-10人 |
| 聞くこと | コンセプトへの反応、使いたいと思うか、懸念点 |

**確認すべきコンセプト**
- 年俸制（月額ではなく年単位で渡す）
- 親預かり型利息（家庭内銀行）
- 家庭内案件（成果報酬型お手伝い）
- 年末レビュー（振り返りの仕組み化）

#### 方法2: ランディングページテスト（スモークテスト / Fake Door Test）

| 項目 | 内容 |
|------|------|
| 準備物 | サービス説明LP + クリック後の簡易質問（メール不要） |
| 方法 | LPを公開し、CTAクリック＋簡易質問で需要を測る |
| 測定 | 訪問数、クリック率、質問回答率 |

**LPに含める内容**
- サービスのコンセプト説明
- 解決する課題
- 主な機能（イメージ画像）
- 1つのメインCTA（クリックで簡易質問に遷移）
- CTAクリック後 → 「30秒の簡易質問」（メール不要）

**トラフィック獲得方法**

| 方法 | コスト | 特徴 |
|------|--------|------|
| SNS投稿（X、Threads） | 無料 | 知人・フォロワー経由、拡散次第 |
| 親向けコミュニティ投稿 | 無料 | ターゲット精度高い |
| TikTok / Instagram Reels | 無料 | バズれば拡散、ただし購入直結しにくい |
| Meta広告（FB/Instagram） | 3,000〜10,000円 | **親層に最も相性良い（推奨）** |
| Google広告 | 1-3万円 | 検索意図のあるユーザー |

### 22.4.1 集客チャネルの役割分担

> このサービスの購買者は「親」。TikTokより**Meta広告の方が狙い撃ちしやすい**。

| チャネル | 役割 | 特徴 |
|----------|------|------|
| **TikTok / Reels** | 認知（知ってもらう） | バズと購入が直結しにくい |
| **X / Threads** | 認知 + 共感 | 親アカウント、教育アカウントにリーチ |
| **Meta広告** | **獲得（登録・購入）** | 子育て・教育関心層を精度高くターゲティング |

**推奨順序**
```
1. まずX/Threadsで無料投稿（反応確認）
2. 反応があればMeta広告に少額投下（3,000〜10,000円）
3. TikTokは補助線として並行運用
```

### 22.4.2 TikTok動画の3つの型

TikTokは「教育サービスの説明」より、**家庭あるある→解決の一撃**が強い。

#### 型A: 親のあるある（共感）→ルール化の解決

```
【15-25秒】
0-2秒:  テロップ「また"買って"って言われた…どう返す？」
3-7秒:  家庭内のワンシーン（再現 or テキスト会話）
8-12秒: 解決「"その場判断"をやめて、年俸とルールで固定する」
13-18秒: 仕組み「年俸を決める→支出を記録→月1レビュー」
19-25秒: CTA「テンプレ配布中（プロフィールのリンク）」
```

#### 型B: 子どもの学び（子視点）

```
【15-20秒】
テーマ: 「今月は我慢したら、欲しいもの買えた」
ポイント: "節約"じゃなく"選択"の体験として見せる
CTA: 「親子で始める年俸制」
```

#### 型C: 破綻→回復（このサービスの独自性）

```
【20-30秒】
Hook: 「使い切ったら終わり…じゃない」
内容: 回復手段がある（案件で稼ぐ/前借り/利息放棄）
ポイント: 責めない、学びの機会として扱う
CTA: 「失敗しても大丈夫な仕組み」
```

#### 動画素材（お金をかけない方法）

| 素材タイプ | 作り方 |
|-----------|--------|
| テキスト会話 | 親子の会話をテロップで表現（実写より作りやすく伸びやすい） |
| 生活小物 | レシート、財布、文房具、机（顔出し不要） |
| 図解 | CapCutで年俸→残高→レビューの流れを3枚で見せる |
| フリーB-roll | 手元、家族の後ろ姿、街並み（顔は出さない） |

**制作ツール**
- CapCut（無料、テロップ・編集）
- Canva（図解・サムネ）
- VOICEVOX（顔出しなしの音声）

**メールリスト活用**
- 登録者にヒアリング依頼（機能要望、課題の深掘り）
- MVP設計へのフィードバック収集
- リリース時の初期ユーザー獲得

**判定基準（Phase 2）**
- ✅ Go: LP転換率 5-10%以上、または「ぜひ使いたい」が50%以上
- ⚠️ 要検討: LP転換率 2-5%、または「使ってみたい」が50%以上
- ❌ Pivot/Stop: LP転換率 2%未満、または反応ゼロ → アイデア再検討

### 22.5 Phase 3: 支払い意思検証

### 22.10 推奨アプローチ（このサービスの場合）

> TikTokも使えるが、最初は**「LP＋単一CTA＋簡易質問」中心**で需要の有無を確認する。
> 理由: まずは「興味があるか」を測る方が母数が取れる。

**推奨の全体フロー**

```
1. LP（1枚）を作る
   ↓
2. フェイクドア（単一CTA）を置く
   ↓
3. 集客
   - まずはX / Threads / 親コミュニティ（無料）
   - 反応が良ければMeta広告に少額投下
   ↓
4. 反応が十分ならPhase 3で価格/有料検証へ進む
```

**なぜこの順序か**

| 順序 | 理由 |
|------|------|
| LP先行 | 「行動」を測れる場所がないと、SNSでもTikTokでも検証にならない |
| 複数CTA | 「どの価値が刺さるか」を同時に測定できる |
| Meta広告 | 親層への精度が高い。TikTokは認知、Metaは獲得 |
| 有料先行 | 「欲しい」と「払う」は別。購入が最強の需要証明 |

> **約2ヶ月で検証完了**。この期間で需要が確認できなければ、開発に着手しないことで無駄な工数を防ぐ。

## Appendix A. 想定Q&A（導入時に必ず出る質問）

> 本章は、制度導入時・運用初期に親子双方から高い確率で発生する質問を想定し、事前に言語化しておくための補足資料。既存ルールの解釈補助を目的とし、新たな義務や制約を追加するものではない。

### A-1. なぜ毎月のお小遣いではなく「年俸制」なのか？

- 毎月制は「今月分を使い切る思考」になりやすい
- 年俸制は「いつ・何に・どれくらい使うか」を自分で配分する思考を促す
- これは節約の訓練ではなく、**選択と結果を引き受ける訓練**である

### A-2. 途中で足りなくなったらどうするの？

- 原則として感情的な追加支給は行わない
- 事前に合意した回復手段（案件、前借り、利息放棄など）から選ぶ
- 生活必需品は常に親負担で確保されるため、生活が破綻することはない

### A-3. 失敗したら可哀想では？

- 本制度では失敗は「起きてはいけないもの」ではない
- 失敗は設計通り起きうるイベントであり、振り返りの材料である
- 叱責や評価ではなく、「次はどうするか」を考えるために扱う

### A-4. 親が管理しすぎにならない？

- 親の役割は監視者ではなく、ルール設計者・監査役
- 日々の使い方に口出しするのではなく、レビュー時に一緒に振り返る
- 見える化は信頼を壊すためではなく、対話を成立させるために行う

### A-5. 他の家庭と比べて少なくない？多くない？

- 本制度において他家庭との比較は行わない
- 年俸額や利率は家庭の価値観・経済状況によって異なってよい
- 比較ではなく「自分の中での判断の質」を評価軸とする

### A-6. お手伝いをお金で釣ることにならない？

- 日常的な家事は原則として報酬対象外
- 家庭内案件は「成果があり、価値を提供した場合」に限定される
- 目的は金額ではなく、価値と対価の関係を理解することにある

### A-7. AIの分析結果はどこまで信用すべき？

- AIの提示はあくまで参考情報であり、正解ではない
- 最終的な判断と合意は必ず親子で行う
- AIは対話を助ける補助輪として扱う

## Appendix B. 用語集（Glossary）

| 用語 | 定義 |
|------|------|
| **年俸** | 1年間で子どもが自由裁量として使える金額の上限 |
| **月俸** | 年俸を月単位に分割して支給する方式 |
| **必須支出** | 親が負担する支出（学校指定品、制服など） |
| **裁量支出** | 子どもが年俸から支払う支出（娯楽、私服など） |
| **手元残高** | 子どもがすぐに使える状態の残高 |
| **親預かり残高** | 親が「家庭内銀行」として預かっている残高 |
| **疑似利息** | 教育目的で設定される、実際の金融商品とは連動しない利息 |
| **固定運用** | 一定期間引き出さないことを条件に追加ボーナスがつく仕組み |
| **家庭内銀行** | 親が子どもの資金を預かり、利息を付与する疑似的な銀行機能 |
| **親ファンド** | 子どもが親に資金を預け、成果条件に応じてリターンを得る仕組み |
| **家庭内案件** | 成果報酬型の「お手伝い」。事前合意と成果確認が必要 |
| **家庭内ビジネス** | 子どもが家族を顧客として行う小規模なビジネス活動 |
| **年末レビュー** | 年度末に行う振り返りと来年度ルール設計の作戦会議 |
| **回復手段** | 資産不足時に利用できる手段（案件、前借り、利息放棄など） |
| **前借り** | 次年度年俸から一部を前倒しで受け取る仕組み |
| **監査役** | 親の役割を表す概念。管理・監視ではなく、ルールの運用確認を行う |
| **スモークテスト（Fake Door Test）** | 実際の製品を作る前にLPや申込フォームで需要を測定するリーンスタートアップ手法。Dropbox、Product Hunt等が活用 |
| **Annual Allowance** | 年俸制お小遣いの英語表現。Pocket Salaryの海外展開時のキーワード |
| **Vercel Analytics** | Vercelが提供する分析ツール。ページビュー・カスタムイベントを追跡 |
| **CTAクリック率** | CTA（行動喚起）ボタンのクリック数 ÷ ページビュー。需要の指標 |
| **i18n（国際化）** | Internationalizationの略。多言語対応の仕組み |
| **Accept-Language** | ブラウザが送信するHTTPヘッダー。ユーザーの言語設定を示す |
